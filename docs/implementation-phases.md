# 実装計画書

## 1. この計画書の目的

この文書は、`unso-os` の実装順を依存関係ベースで整理し、未確定事項を先に見える化するためのものです。  
現時点では、認証の入口と初期スキーマはありますが、業務利用の開始条件と運用ルールが未確定なため、実装順を誤ると手戻りが大きくなります。

## 2. 現在の前提

- Google ログインとセッション維持は実装済み
- `companies`、`employees`、`emp_requests` を含む初期スキーマは存在する
- `company_id` ベースの RLS はあるが、ロール別制御は未整備
- 業務画面、業務 API、社員導線、初回利用開始導線は未整備
- `/` は仮ページのため、本番の業務開始地点はまだない

## 3. 未確定事項一覧

### 3-1. 開発環境

| 論点 | 現状 | 決めること | 放置した場合の影響 |
|---|---|---|---|
| Node / npm の基準バージョン | 文書化なし | 推奨バージョンを固定するか | メンバーごとに起動差異が出る |
| 必須環境変数 | `.env.local` 前提だが一覧なし | 必須キーと取得元を明文化するか | ローカル起動で詰まりやすい |
| Supabase 接続先 | 接続土台のみ | 開発用と本番用の切り分け方 | 誤って本番データを触る恐れ |
| 初期データ投入 | `expense_categories` などの投入方法未定 | 手動投入か自動投入か | 画面だけ作っても動作確認できない |
| テスト方法 | `lint` はあるが、確認フロー未定 | 最低限の動作確認手順をどこまで標準化するか | 実装者ごとに確認品質がばらつく |

### 3-2. デプロイ運用

| 論点 | 現状 | 決めること | 放置した場合の影響 |
|---|---|---|---|
| Vercel 管理者 | 文書化なし | 誰が本番設定を持つか | 障害時に対応できない |
| 本番環境変数管理 | 方針未記載 | どこで管理し、誰が更新するか | デプロイ後に認証や接続が壊れやすい |
| 公開前チェック | 手順未定 | 最低限の確認項目を何にするか | 不完全な状態で公開しやすい |
| バックアップ / 切り戻し | 方針未定 | Supabase と Vercel で何を戻せるようにするか | 障害時の復旧判断が遅れる |

### 3-3. 社員導線

| 論点 | 現状 | 決めること | 放置した場合の影響 |
|---|---|---|---|
| ログイン後の分岐 | 認証後は一律 `/` へ遷移 | 有効社員 / 未登録 / 申請中の分岐先 | ログインしても使い始められない |
| 最初の会社作成 | テーブルはあるが導線なし | 誰がどこで会社を作るか | SaaS 初回導入を開始できない |
| 最初の `OWNER` 作成 | 仕組み未実装 | 自動作成か手動承認か | 管理者不在で運用開始できない |
| 社員申請方式 | `emp_requests` はある | 申請制、招待制、併用のどれにするか | UI と API の設計が定まらない |
| 承認待ち画面 | 未実装 | 申請中ユーザーに何を見せるか | 利用者が状況を把握できない |

### 3-4. マルチテナント

| 論点 | 現状 | 決めること | 放置した場合の影響 |
|---|---|---|---|
| `employees` と Auth の紐付け | `google_email` ベース | メール変更時や重複時の扱い | 誤認証や運用停止の原因になる |
| RLS の責務範囲 | 会社分離中心 | ロール別更新制御を DB 側でどこまで持つか | 画面と API だけでは漏れが出る |
| company 作成時の初期処理 | 未定 | マスタ初期値と初期社員を何まで自動生成するか | 導入作業が属人化する |
| 共通設定と会社別設定の境界 | 方針未定 | 将来の業種設定をどこで持つか | 後から抽象化コストが増える |

### 3-5. マスタ設計

| 論点 | 現状 | 決めること | 放置した場合の影響 |
|---|---|---|---|
| どのマスタが業務開始に必須か | 未定 | 最低限の初期マスタセット | 入力画面を先に作っても運用できない |
| 荷主・ルート・運賃の責務 | テーブルのみ存在 | どの単位で紐付けるか、必須入力は何か | 運行実績画面の仕様がぶれる |
| 経費区分初期値 | 旧 GAS には候補あり | 自動投入の有無と編集責任 | 経費申請の初回体験が不安定になる |
| 他業種抽象化の境界 | ラベル化のヒントあり | 表示文言だけで足りるか、データ構造変更が必要か | 過剰抽象化か再設計が発生する |

## 4. 先に決めるべき事項

優先度の高い順に、次の 5 点を先に決める必要があります。

1. ログイン後のユーザー分岐
2. 最初の会社作成と最初の `OWNER` 作成方法
3. 社員申請フローの運用方式
4. ロール制御を画面 / API / RLS のどこまで持つか
5. 業務開始に必要な最小マスタ

関連文書:

- Phase 1 の具体仕様案は `docs/phase-1-auth-onboarding-spec.md`
- 上記文書で、ログイン後分岐、初回会社作成、初回 `OWNER`、`company_code`、再申請方式の方針を確定済み

## 5. 実装フェーズ

### Phase 0: 開発環境と外部サービスの可視化

目的:

- ローカル起動条件と外部サービス依存を見える化する

作業内容:

- 必須環境変数一覧を作る
- Supabase / Google OAuth / Vercel の管理責任者を明確にする
- 初期データ投入方法と最低限の動作確認手順を決める

このフェーズが先な理由:

- 実装を進めても、他メンバーが再現できなければ検証が止まるため

### Phase 1: 認証後導線と初回利用開始導線

目的:

- ログインしたユーザーが「使える」「申請中」「未登録」を判別できるようにする

作業内容:

- 認証後の分岐ルールを決める
- 初回会社作成と初回 `OWNER` 作成の方式を決める
- 社員申請、承認待ち、利用開始の最小画面を定義する
- 詳細仕様は `docs/phase-1-auth-onboarding-spec.md` に切り出して管理する

依存:

- Phase 0 の環境情報

このフェーズが最優先な理由:

- 今はログインできても業務開始できないため、全機能の入口が欠けている

### Phase 2: 権限とマルチテナント責務の固定

目的:

- ロール制御と会社分離の責務を、画面・API・DB でそろえる

作業内容:

- `DRIVER` / `ADMIN` / `OWNER` ごとの操作範囲を定義する
- API で必要な権限チェックを整理する
- RLS で company 単位とロール単位のどこまで担保するか決める
- `employees` と認証ユーザーの紐付けルールを固める

依存:

- Phase 1 の利用開始導線

このフェーズが先な理由:

- 業務機能を先に作ると、後から権限差し替えで大きく作り直すため

### Phase 3: 共通マスタと初期データの整備

目的:

- 業務画面を成立させるための前提データをそろえる

作業内容:

- 荷主、ルート、車両、運賃、経費区分の責務を定義する
- company 作成時の初期投入対象を決める
- 他業種展開に備え、ラベルと初期値の抽象化境界を定める

依存:

- Phase 2 の権限・会社責務

このフェーズが先な理由:

- マスタ未確定のまま業務画面を作ると、入力項目と登録条件がすぐ崩れるため

### Phase 4: 運行実績 MVP

目的:

- 実際に日々使うコア業務を最初に成立させる

作業内容:

- `/` を運行実績入力の本番トップとして再設計する
- 実績登録、一覧、承認の最小導線を作る
- 運賃やルートとの参照関係を固める

依存:

- Phase 3 の共通マスタ

このフェーズを先に置く理由:

- ルートディレクトリの想定役割と旧システムの中心業務に最も近く、運用価値が高いため

### Phase 5: 経費・勤怠

目的:

- 運行実績に続く日次業務を横展開する

作業内容:

- 経費申請と承認フローを実装する
- 勤怠入力と承認フローを実装する
- 共通する申請・承認 UI の部品化方針を整理する

依存:

- Phase 2 の権限
- Phase 3 のマスタ

この順序にする理由:

- 運行実績より業務開始地点との結び付きが弱いため、2 段目で十分

### Phase 6: ダッシュボード・運用補強

目的:

- 経営者向けの閲覧と公開運用を安定させる

作業内容:

- `OWNER` 向けダッシュボードの範囲を決める
- 公開前チェックリストを整備する
- 障害時の切り戻し方針を定義する

依存:

- Phase 4 と Phase 5 の業務データ蓄積

この順序にする理由:

- ダッシュボードは入力データがないと価値が出にくく、後段のほうが判断しやすいため

## 6. 優先順位まとめ

実装優先順位は次の順を推奨します。

1. 使い始められる状態を作る
2. 権限事故を防ぐ
3. マスタ不足をなくす
4. 運行実績を先に成立させる
5. 経費と勤怠を追加する
6. ダッシュボードと公開運用を整える

## 7. フェーズごとの確認ポイント

- Phase 0 完了条件: 別環境でも起動条件を再現できる
- Phase 1 完了条件: ログイン後にユーザー状態ごとの画面分岐ができる
- Phase 2 完了条件: ロール違反操作を画面・API・DB のいずれでも防げる
- Phase 3 完了条件: 業務画面に必要な最低限マスタが定義済み
- Phase 4 完了条件: 運行実績の登録から承認まで最小導線で通せる
- Phase 5 完了条件: 経費と勤怠が同じ考え方で申請・承認できる
- Phase 6 完了条件: 公開判断に必要な運用情報がそろう
