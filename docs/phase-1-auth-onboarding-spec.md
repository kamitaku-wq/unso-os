# Phase 1 仕様

## 1. この文書の対象

この文書は、Phase 1 で先に固めるべき次の 2 点を具体化した確定仕様です。

- ログイン後の分岐
- 最初の会社作成と最初の `OWNER` 作成

目的は、Google ログイン後にユーザーが迷わず次の画面へ進めるようにし、同時に SaaS 初回導入の開始条件を明確にすることです。

## 2. 先に置く判断

この仕様では、次の判断を採用します。

1. Google ログイン成功だけでは業務利用開始にしない
2. 認証直後は専用の分岐ルートでユーザー状態を判定する
3. システム全体で会社が 1 件もないときだけ、初回セットアップを許可する
4. 会社が存在する場合の新規ユーザーは、対象会社を指定して社員申請を行う
5. 社員申請には `company_id` を特定するための `company_code` を使う
6. `company_code` は初回セットアップ時に自動生成する
7. 却下済み申請は履歴を残したまま新規申請を再作成する
8. 初期段階の `ADMIN` 追加は社員申請の承認で行う
9. 初回セットアップ完了後は `/dashboard` に遷移する

## 3. 追加する概念

### 3-1. `company_code`

社員申請時に「どの会社へ参加するか」を指定するため、会社ごとに人が入力しやすい参加コードを持たせます。  
`company_code` はユーザー手入力ではなく、初回セットアップ時にシステムが自動生成します。

用途:

- 社員申請画面で所属先会社を特定する
- 初回会社作成時に一緒に発行する
- 招待リンクが未実装でも、最小運用を開始できるようにする

要件:

- 会社ごとに一意（重複しない）
- 半角英数字とハイフン程度に制限する
- 画面で共有しやすい長さにする
- `OWNER` があとで確認・再共有できる

生成方針:

- 会社名をもとにした短い接頭辞 + ランダム文字列で生成する
- 重複時は再生成する
- 初回は自動生成値を表示し、その値で作成する

補足:

- 将来的に招待リンク方式へ広げてもよいが、初期段階では `company_code` のほうが実装が単純です

### 3-2. 認証後分岐ルート

Google OAuth コールバックの直後に `/auth/post-login` へ集約し、そこでユーザー状態を判定します。

理由:

- `/` に直接飛ばすと、未登録ユーザーと有効社員を分けにくい
- 将来ルート別画面が増えても、判定ロジックを 1 か所にまとめられる

## 4. ログイン後の状態定義

認証済みユーザーは、次の 5 状態のいずれかに分類します。

| 状態 | 判定条件 | 遷移先 |
|---|---|---|
| `BOOTSTRAP_REQUIRED` | `companies` が 0 件 | `/setup` |
| `ACTIVE_EMPLOYEE` | `employees` に対象ユーザーが存在し `is_active = true` | ロール別画面 |
| `PENDING_REQUEST` | `emp_requests` に `PENDING` がある | `/pending` |
| `REJECTED_OR_INACTIVE` | 申請却下済み、または `employees.is_active = false` | `/register` |
| `UNREGISTERED` | 上記いずれでもない | `/register` |

## 5. ログイン後の分岐仕様

### 5-1. 画面遷移

Google ログイン成功後は、次の順で判定します。

1. `companies` 件数を確認する
2. 0 件なら `/setup` にリダイレクトする
3. 1 件以上なら、認証ユーザーのメールで `employees` を確認する
4. 有効社員ならロール別画面へ送る
5. 有効社員でなければ `emp_requests` を確認する
6. 承認待ちなら `/pending`、それ以外は `/register` に送る

### 5-2. ロール別遷移先

| ロール | 第1遷移先 | 補足 |
|---|---|---|
| `DRIVER` | `/` | 将来の運行実績入力トップ |
| `ADMIN` | `/admin` | 承認とマスタ管理の入口 |
| `OWNER` | `/dashboard` | 経営ダッシュボードの入口 |

暫定運用:

- 実ページ未実装の間は、未実装ページから `/` の共通ハブへフォールバックしてよい
- ただし、判定ロジック自体は最初からロール別に持つ

### 5-3. 各画面の役割

| パス | 役割 |
|---|---|
| `/auth/post-login` | 認証直後の状態判定だけを行う |
| `/setup` | 最初の会社と最初の `OWNER` を作る |
| `/register` | 既存会社への社員申請を送る |
| `/pending` | 承認待ち状態を表示する |
| `/` | `DRIVER` 向け業務開始画面 |
| `/admin` | `ADMIN` 向け承認・管理画面 |
| `/dashboard` | `OWNER` 向け閲覧画面 |

## 6. 最初の会社と最初の `OWNER` 作成仕様

### 6-1. 実行条件

初回セットアップは、次の条件をすべて満たすときだけ実行できます。

- 認証済みである
- `companies` が 0 件である
- 同時実行チェックで、まだ他ユーザーが作成完了していない

### 6-2. 入力項目

`/setup` では、最低限次を入力します。

- 会社名
- 初回 `OWNER` 表示名

初回ログインメールは認証済み情報から自動取得し、手入力させません。  
`company_code` は入力項目ではなく、確認用の表示項目とします。

### 6-3. 作成内容

セットアップ成功時に、次を 1 トランザクション（途中失敗ならまとめて取り消す処理）で作成します。

1. `companies` に会社を作成する
2. ログイン中ユーザーを `employees` に `OWNER` / `is_active = true` で作成する
3. `expense_categories` の初期値を投入する
4. 初回セットアップ完了フラグとして利用できる状態を返す

初期投入の対象:

- まずは `expense_categories` のみを必須とする
- `customers`、`routes`、`vehicles`、`ratecards` は空で開始してよい

### 6-4. セットアップ後の遷移

初回セットアップ完了後は `/dashboard` に送ります。

理由:

- 初回 `OWNER` は会社全体設定と社員導線の確認が必要なため
- 運行実績入力よりも、導入直後の管理作業を優先すべきため

## 7. 既存会社への社員参加仕様

### 7-1. 前提

`emp_requests.company_id` は必須なので、申請時に対象会社を特定する必要があります。  
そのため、`/register` では `company_code` を入力させ、バックエンドで `company_id` を解決します。

### 7-2. 入力項目

- 氏名
- 希望ロール
- `company_code`

ログインメールは認証済み情報を使います。

### 7-3. 登録ルール

- `company_code` が見つからなければエラー
- 同一メールの `PENDING` 申請が同一会社に存在すれば再申請不可
- 承認時は `employees` に自動登録し、有効化する
- 却下済みの場合は既存レコードを上書きせず、新しい `emp_requests` を追加して再申請する
- 過去の `REJECTED` は履歴として残す

### 7-4. `ADMIN` 追加方法

初期段階では、`ADMIN` の追加は社員申請を `OWNER` または既存 `ADMIN` が承認する方式に固定します。

この時点で採らないもの:

- `OWNER` からの直接招待
- メール招待リンク
- 管理画面からの即時ユーザー発行

理由:

- Phase 1 で画面と API を増やしすぎないため
- まずは `emp_requests` を中心にした最小運用を成立させるため

## 8. 権限ルール

### 8-1. `/setup`

- `companies = 0` のときのみ表示可
- 会社が 1 件でも存在したら表示不可
- 認証済みユーザーのみ実行可

### 8-2. `/register` と `/pending`

- 未有効社員の認証済みユーザーのみ利用可
- 有効社員が開いた場合は `/auth/post-login` へ戻す

### 8-3. ロール別画面

- ロール不一致の場合は表示しない
- 画面制御だけでなく API と RLS でも拒否する

## 9. 実装順

この仕様に基づく実装順は次を推奨します。

1. `/auth/post-login` の状態判定
2. `/setup` の初回会社作成
3. `company_code` を前提にした `/register`
4. `/pending`
5. ロール別の遷移先ページの暫定ハブ化

## 10. Phase 1 で確定した決定事項

この仕様で Phase 1 として確定した決定事項は次のとおりです。

- `company_code` は初回セットアップ時に自動生成する
- 却下済み申請は履歴を残し、新規レコードで再申請する
- `ADMIN` 追加は社員申請の承認で行う
- 初回セットアップ完了後の遷移先は `/dashboard` に固定する

将来検討:

- `OWNER` からの直接招待機能
- `company_code` の再発行 UI
- セットアップ完了後に `master` 初期設定へ誘導する導線強化
