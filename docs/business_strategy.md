# 運送OS 事業展開・開発プラン

> 作成：2026年3月
> 目的：事業の方向性・技術方針・フェーズ別のアクションを一元管理する

---

## 全体方針（3行）

- **技術モデルはC（コアとアドオンを分ける SaaS）** で進める
- **最初から業種を絞らない**。運送業を第1実装とし、他業種も初期から受け入れる。差分を記録しながらコアとモジュールを育てていく
- **AI全開発を前提**とする。設計の意思決定だけは人間が持ち続ける

---

## 展開モデルの選択理由

### 検討した3モデル

| モデル | 概要 | 採用判断 |
|--------|------|---------|
| **A（標準 SaaS）** | 全社共通の機能・画面。設定のみカスタマイズ | 差分吸収力が低い。業種展開不可 |
| **B（フォーク受託）** | 顧客ごとにコードをコピーして個別改修 | 10社で作業量が破綻。スケールしない |
| **C（コア + アドオン）** | 共通部分はひとつ。業種・帳票だけ差し替え可能 | **採用** |

### モデルCを選んだ理由

- バグ修正・アップデートが全社に一括反映される
- インフラコストが顧客数に比例して増えない（1プロジェクトで全社管理）
- コアと業種を分離する設計が、AI開発の精度を保つ構造とも一致する
- `company_id` の一貫性により、将来の安全な分離移行が可能

### コスト比較（10社時点）

| | モデルB | モデルC |
|-|--------|--------|
| 開発者時間（累計） | 380〜1,350時間 | 180〜430時間 |
| Supabase/月 | $250 | $25 |
| Vercel/月 | $200 | $20 |
| **インフラ合計/月** | **$450** | **$45** |

## フェーズ別プラン

### フェーズ1：基盤固め（0→3社）※現在

**目標**：複数業種を受け入れながら実際に使ってもらい、コアと業種固有の切り分けを実地で学ぶ

#### 技術タスク

| 優先度 | タスク | 内容 |
|--------|--------|------|
| 最高 | ディレクトリ構成の整理 | `lib/core/` と `lib/industries/transport/` に分離する |
| 最高 | companiesテーブルの拡張 | `industry` と `custom_settings` カラムを追加 |
| 高 | 未完成機能の完成 | 社員申請承認フロー・請求書生成・月次締め |

#### ビジネスタスク

- 業種を問わず最初の1〜3社を**無償モニター**として獲得し、実業務で使ってもらう
- 差分ログ（`docs/customer_diff_log.md`）に業種・顧客ごとの「違い」を記録し続ける
- 料金プランの仮設計を行う

#### フェーズ2移行の判断基準

```
✅ 1社以上が実業務で3ヶ月以上使い続けている
✅ 差分ログに10件以上の「要望・違い」が記録されている
✅ 主要機能（実績・経費・勤怠・マスタ・承認）が完成している
```

---

### フェーズ2：標準化・有料化（3→10社）

**目標**：有料顧客を獲得し、差分の多い箇所を設定で吸収できる仕組みに変換する

#### 料金プラン（目安）

| プラン | 月額 | 対象 | 機能 |
|--------|------|------|------|
| Standard | ¥15,000 | 10名以下 | 実績・経費・勤怠・マスタ |
| Professional | ¥30,000 | 30名以下 | Standard + 請求書・給与・ダッシュボード |
| Enterprise | 要相談 | 大規模・カスタム要件 | 全機能 + 個別対応 |

> インフラコスト（Supabase Pro + Vercel Pro ≒ ¥7,000/月）は3社で回収できる設計

#### 技術タスク

- **差分の設定化**：フェーズ1の差分ログを元に `custom_settings` で吸収できるものを実装
- **帳票テンプレート化**：`lib/industries/transport/templates/` に顧客別テンプレートを配置
- **最低限の自動テスト追加**：金額計算・承認フローのロジックにテストを追加

#### フェーズ3移行の判断基準

```
✅ 有料顧客が5社以上
✅ 月額収益がインフラ + 開発コストをカバーしている
✅ 差分ログを見て「業種展開すべきか・運送業を深掘りすべきか」の判断材料がある
```

---

### フェーズ3：業種展開準備（10→30社）

**目標**：2業種目の展開判断をデータで行い、コアとモジュールの境界を正式確定させる

#### 技術タスク

- `lib/core/` と `lib/industries/` の正式分離（ロジックレベルで完全に分ける）
- 2業種目のモジュール追加（業種が決まってから設計）
- 将来の移行スクリプト準備（1社を別プロジェクトに安全に移せるSQL）

#### ビジネスタスク

- 差分ログと問い合わせ実績から2業種目を選定
- 業種ごとの料金体系を設計
- 販売代理店・紹介者の開拓

#### フェーズ4移行の判断基準

```
✅ 有料顧客が15社以上
✅ 2業種目の顧客候補が複数いる
✅ AIだけでのメンテナンスが継続できている
```

---

### フェーズ4：分離・スケール（30社〜）

**目標**：業種・規模に応じた技術分離を実施し、事業をスケールさせる

#### 技術の分離判断

| 分離の種類 | タイミング | 内容 |
|-----------|-----------|------|
| 業種ごとにVercelプロジェクトを分ける | 業種が3つ以上 | コードの独立性を高める |
| 大口顧客を専用 Supabase プロジェクトへ | 月額¥100万以上の顧客 | SLA保証・完全カスタマイズ |
| 全社一括→業種別マルチテナントへ移行 | 30社前後 | 安全に移行できる（設計が整っていれば） |

#### 移行が安全にできる条件

```
条件1：company_id が全データで一貫している（現在OK）
条件2：コアと業種固有の境界がコード上で明確になっている
条件3：移行前後で同じデータが取得できるかテストできる
```

## インフラのスケール計画

### Supabase の限界

| プラン | DB容量 | 目安顧客数（運送業1社=5MB/年） | 月額 |
|--------|--------|---------------------------|------|
| Free | 500MB | 〜100社×1年分 | $0 |
| Pro | 8GB | 〜1,600社×1年分 | $25 |

## AI全開発の限界と対策

### 限界が来る順番

| 限界の種類 | 深刻度 | タイミング |
|-----------|--------|----------|
| Supabaseのデータ容量 | 低 | 実質問題にならない |
| インフラコスト | 中 | 顧客増加とともに上昇（予測可能） |
| コード量による AI 精度低下 | 高 | 15,000行〜（現在の約2倍） |
| 指示品質・設計書の陳腐化 | 最高 | 今すぐから継続的に対処が必要 |

### AI開発を維持するための永続ルール

| ルール | 理由 |
|--------|------|
| CLAUDE.md を常に最新に保つ | AIの精度を保つ唯一の方法 |
| 1ファイル200行以内を守る | AIが一度に把握できる量を超えさせない |
| 新機能を追加したらドキュメントに記録 | 次のAIセッションで文脈を失わない |
| DBスキーマの変更はマイグレーションファイルで | 直接編集すると追跡不能になる |
| 差分ログを書き続ける | 業種展開の判断材料になる |

```
□ 業種を問わずモニター顧客候補をリストアップする
```
