# AI実装指示書

## 1. この指示書の目的

この文書は、`unso-os` を実装するときに AI が守る判断基準をそろえるためのものです。  
対象は MVP だけではなく、将来の SaaS 展開まで見据えた全体設計です。ただし、初期段階では監査や証跡は最小限とし、まずは現場で使い始められる実務導線を優先します。

## 2. 前提整理

- 現時点で実装済みなのは、Google ログイン、Supabase 接続、初期 DB スキーマ、`company_id` ベースの RLS まで
- `/` はまだ仮トップのため、ログイン後の業務開始地点は未整備
- Phase 1 の認証後導線、初回会社作成、初回 `OWNER`、`company_code` 方針は `docs/phase-1-auth-onboarding-spec.md` で確定済み
- 旧 GAS は参照元ではあるが、そのまま正解源として固定せず、現行リポジトリとユーザー確認を合わせて判断する
- 運送業を標準にしつつ、他業種に広げる余地を残す

## 3. 参照優先順位

仕様がぶつかったときは、次の順で確認します。

1. ユーザーがこの会話で明示した最新指示
2. 現行リポジトリ内の実装とドキュメント
3. 旧 GAS の設計・ロジック
4. 一般的な実装慣習

補足:

- 現行コードと旧 GAS が食い違う場合は、旧実装をそのまま移植せず、業務意図を確認してから決める
- 権限や承認条件のように業務影響が大きい箇所は、推測で埋めない

## 4. 章立て

### 4-1. プロダクト前提

- 何を作るか: 運送業向けを標準とするマルチテナント SaaS
- 誰が使うか: `DRIVER`、`ADMIN`、`OWNER`
- 何が未実装か: 業務画面、業務 API、ロール別制御、社員導線、初期データ投入

### 4-2. 判断ルール

- 画面、API、RLS の 3 層で同じ権限制御をそろえる
- ログインできても社員未登録なら業務画面へ入れない前提で考える
- 会社作成と初回オーナー作成は、通常の社員申請フローと分けて扱う
- 運送業固有の表示文言は設定で差し替え可能にし、業務データ構造まで安易に業種固定しない
- 既存 DB に列があっても、運用ルールが未確定なら画面や API を先に断定しない

### 4-3. 実装原則

- まずローカルで再現できる導線から作る
- 1 つの機能を作るときは、画面だけで終わらせず、必要な API と DB 制約まで合わせて考える
- `company_id` 分離は UI ではなく DB 側で担保し、UI は補助にとどめる
- 初期導入で必要な最小マスタを決めてから業務入力画面を作る
- 他業種対応は「ラベル」「初期マスタ」「設定値」の抽象化を優先し、承認フローや会計ルールまで最初から共通化しすぎない

### 4-4. 確認必須項目

次の項目は、実装前に確認または明示が必要です。

- 認証: Google ログイン後の分岐を `docs/phase-1-auth-onboarding-spec.md` に従う
- 社員導線: Phase 1 では `company_code` + 社員申請方式を採用し、招待制は将来検討とする
- 会社作成: 初回は `/setup` から会社作成する
- 初回オーナー: 初回セットアップ時に認証ユーザーを `OWNER` として作成する
- マスタ初期値: `expense_categories` などを company 作成時に自動投入するか
- 外部サービス: Supabase、Google OAuth、Vercel の権限保有者と管理方法
- 環境変数: 必須キー、保管場所、ローカル共有方法

### 4-5. 禁止事項と注意事項

- `.env.local` をコミットしない
- 既存の適用済みマイグレーションを直接編集しない
- 権限制御を推測で実装しない
- 他社データ分離をアプリ側条件分岐だけで済ませない
- 旧 GAS の都合で存在した挙動を、理由確認なしに踏襲しない
- 仮画面を本番導線として固定化しない

## 5. 必須判断ルール

### 5-1. 認証と利用開始

- Google 認証成功と業務利用開始は別物として扱う
- `employees` に有効な行がないユーザーは、入力画面ではなく利用開始導線へ送る
- 社員申請が承認待ちなら、申請状況を確認できる待機画面を出す

### 5-2. 権限

- `DRIVER` は自分のデータ入力と自分の履歴閲覧を基準にする
- `ADMIN` は承認、差戻し、マスタ管理を担当する
- `OWNER` は全権限に加えて、全体閲覧と経営ダッシュボードを担当する
- 画面でボタンを隠すだけでなく、API と RLS でも拒否する

### 5-3. マルチテナント

- 全業務データは `company_id` を基準に分離する
- 会社横断で共有する設定を作る場合でも、共有理由と管理責任を明記する
- company 作成時に必要な初期データ投入の責務を明確にする

### 5-4. マスタ設計

- 荷主、ルート、車両、運賃、経費区分は会社単位マスタを基本にする
- 他業種対応は、まず表示名と初期データで吸収できるかを確認する
- 業務ごとに必須になるマスタが未整備なら、その業務画面は後ろ倒しにする

### 5-5. 実装順

- 最優先はログイン後導線と社員導線
- 次にロール制御とマスタ責務
- その後に運行実績、経費、勤怠などの業務機能を依存関係順に実装する

## 6. AI が迷ったときの扱い

- 認証、権限、会社作成、社員導線、初期データ投入に関わる曖昧さは確認を優先する
- UI 文言や軽微なレイアウトは、既存トーンに合わせて AI が自律判断してよい
- 実装コストより運用事故リスクが大きい箇所は、保守的な案を採る
- 複数案がありうるときは、メリット・デメリット付きで短く提示する
