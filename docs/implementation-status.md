# 運送OS 現状整理

## 1. プロダクトの目的

`unso-os` は、運送会社の日々の業務を 1 つにまとめる SaaS（複数社で共通利用できる業務システム）です。  
旧 GAS 版で扱っていた運行実績、経費、勤怠、マスタ管理、承認業務を、Next.js + Supabase ベースで再構築する前提になっています。

このプロダクトが目指しているのは、次の 3 点です。

- 現場の入力業務を Web で一元化する
- 管理者の承認やマスタ管理をオンラインで回せるようにする
- 会社ごとのデータを分離した状態で複数社へ提供できるようにする

## 2. 想定利用者

想定利用者は、運送会社の社内メンバーです。権限（できることの範囲）は 3 ロールで整理されています。

- `DRIVER`: 現場作業者。自分の実績、経費、勤怠を入力する想定
- `ADMIN`: 管理者。申請承認、マスタ管理、業務チェックを担当する想定
- `OWNER`: 経営者。全体閲覧や意思決定に使う想定

また、SaaS としては「導入企業ごと」にデータを分ける前提です。1 つの Supabase プロジェクトで複数社を扱い、`company_id` と RLS（DB 側の閲覧制御）で他社データを見えなくする設計です。

## 3. 現状実装済みの範囲

### 3-1. 画面

現時点で確認できるページは最小限です。

- `/login`: Google ログイン画面
- `/auth/callback`: OAuth コールバック処理
- `/`: ログイン後の遷移先。ただし中身は Next.js 初期テンプレートのまま

つまり、業務画面そのものはまだほぼ未着手で、「ログインの入口だけ先にある」状態です。

### 3-2. 認証

認証の土台はすでにあります。

- `app/login/page.tsx`
  - Google OAuth でサインインを開始
- `app/auth/callback/route.ts`
  - Supabase から返る認証コードをセッションへ交換
- `proxy.ts`
  - 各リクエストでセッションを更新
  - 未ログイン時は `/login` にリダイレクト
- `lib/supabase/browser.ts`
  - ブラウザ側 Supabase クライアント
- `lib/supabase/server.ts`
  - サーバー側 Supabase クライアント

このため、「Google でログインし、未ログインなら弾く」ところまでは土台があります。

### 3-3. DB 土台

初期スキーマは `supabase/migrations/001_initial_schema.sql` にまとまっています。主な土台は次のとおりです。

- テナント管理: `companies`
- ユーザー管理: `employees`, `emp_requests`
- 業務マスタ: `customers`, `routes`, `ratecards`, `vehicles`, `expense_categories`
- 業務データ: `billables`, `expenses`, `attendances`

さらに、DB ではマルチテナント前提の構造が入っています。

- 主要テーブルに `company_id` を配置
- `employees.google_email` から自分の `company_id` を引く `get_my_company_id()` 関数を定義
- RLS を有効化し、同じ `company_id` のデータだけ見える方針を設定

この段階では「業務データを置く箱」はほぼ揃っています。

## 4. 現時点の実装レベル

全体としては、完成済みの業務システムではなく、初期フェーズです。

- 画面: ログイン導線のみ
- 認証: Google ログインとセッション維持のみ
- DB: 業務テーブルとテナント分離の基礎のみ

逆に言うと、運用で使うための業務画面、API、業務ロジック、権限制御はこれから実装する余地が大きい状態です。

## 5. まだ空いている実装

### 5-1. 未実装の業務画面

プロジェクト説明では次の画面が想定されていますが、現時点では確認できません。

- 管理者承認画面
- 経営ダッシュボード
- 経費申請画面
- マスタ管理画面
- 勤怠管理画面
- 運行実績入力画面としての本番トップ画面

特に `/` は仮トップのままなので、ログイン後にユーザーが実務を始める画面が未整備です。

### 5-2. 未実装の API

`app/api/*/route.ts` 配下の業務 API はまだ存在していません。  
そのため、次のような処理は DB テーブルがあってもまだ実行口（呼び出し窓口）がありません。

- 運行実績の登録・承認
- 経費申請の登録・承認
- 勤怠の登録・承認
- 社員申請の受付・承認
- 荷主、ルート、車両、運賃などのマスタ更新

### 5-3. 未実装の権限制御

今ある権限制御は「ログインしているか」と「同じ会社か」が中心です。  
一方で、業務で必要になる細かい権限制御はまだ未整備です。

- `DRIVER` / `ADMIN` / `OWNER` ごとの画面表示制御
- API ごとのロールチェック
- 承認系操作を `ADMIN` 以上に限定する制御
- `DRIVER` が自分以外のデータを触れない制御
- `OWNER` 向け集計や全体閲覧の制御

### 5-4. DB 運用面のギャップ

RLS の土台はありますが、業務運用には追加整理が必要です。

- 現状のポリシーは会社単位の分離が中心で、ロール単位の制御は未反映
- 書き込み時の `WITH CHECK` を含む詳細ポリシーが未整備で、実際の登録・更新ルールはまだ固まっていない
- `employees` にユーザーをどう登録し、初回ログインをどう許可するかの運用フローが未実装

## 6. 次に実装優先度が高いもの

優先度が高いのは次の順です。

1. ログイン後に出す本番トップ画面を作る
2. 運行実績、経費、勤怠の入力 API を作る
3. `ADMIN` / `OWNER` の承認フローを作る
4. 画面・API・RLS の 3 層でロール制御を揃える
5. 社員登録と初回利用開始の流れを決める
