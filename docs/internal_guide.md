## 1. はじめに

### このシステムは何か

**運送OS（unso-os）** は、運送会社の日々の業務を Web ブラウザ一本で管理するための業務システムです。

具体的には、以下の業務をオンラインで完結できるようにすることを目指しています。

- 運行実績の入力と承認
- 経費申請の入力と承認
- 勤怠の記録と確認
- 荷主・ルート・車両などのマスタ（基本情報の登録一覧）管理
- 経営ダッシュボード（売上・コスト・人件費などの一覧）
- 請求書・給与計算

### 誰が使うか

運送会社の社内メンバーが使うシステムです。権限レベル（ロール）によって、できることが異なります。

| ロール | 対象者 | できること |
|--------|--------|------------|
| DRIVER（ドライバー） | 現場の作業者 | 自分の実績・経費・勤怠の入力 |
| ADMIN（管理者） | 事務担当者など | 申請承認・マスタ管理・業務チェック |
| OWNER（経営者） | 会社オーナー | 全権限 + 経営ダッシュボード閲覧 |

### SaaS としての設計

このシステムは **SaaS（Software as a Service）** として設計されています。
SaaS とは「1つのシステムを複数の会社で共有利用するサービス形態」のことです。

将来的に複数の運送会社へ提供する前提で、各社のデータは完全に分離して管理されます。

### このドキュメントの使い方

- **初めてシステムに関わる人** → 1〜7章を順番に読む
- **開発作業をしたい人** → 5章（環境構築）→ 13章（よくある作業）を参照
- **課金・プラン変更の判断が必要な人** → 2・3・4・16章を参照

## 3. Supabase 詳細

### Supabase とは

**Supabase** は、データベース（PostgreSQL）・認証・ファイルストレージなどをまとめて提供するクラウドサービスです。
このシステムでは「データを保存する場所」と「ログイン管理」の両方を Supabase に頼っています。

### Free プランの制限

| 項目 | 無料枠 | 備考 |
|------|--------|------|
| データベース容量 | 500 MB | テーブル・レコード全体の合計 |
| ストレージ（ファイル保存） | 1 GB | 領収書画像など |
| 月間アクティブユーザー (MAU) | 50,000 人 | ログインしたユーザー数 |
| Edge Functions（サーバー関数） | 500,000 回/月 | 現在は未使用 |
| プロジェクトの一時停止 | あり | 7日間アクセスなしで自動停止（有料プランは停止なし） |

> **重要**: Free プランでは、7日間アクセスがないとプロジェクトが自動停止します。
> 停止した場合は Supabase 管理画面（dashboard.supabase.com）からプロジェクトを再起動できます。

### Pro プランへの移行が必要なタイミング

以下のいずれかに該当したら Pro プラン（$25/月〜）への移行を検討してください。

- データ量が 500MB に近づいてきた
- 月間ログインユーザー数が 50,000 人を超えそう
- 自動停止を避けたい（常時稼働が必要）
- チームメンバーと管理画面を共有したい

### Supabase 管理画面の操作

1. [https://supabase.com/dashboard](https://supabase.com/dashboard) にアクセス
2. Google アカウントでログイン
3. プロジェクト（unso-os）を選択

主な操作場所：

| 操作 | 場所 |
|------|------|
| テーブルの中身を確認 | Table Editor（左メニュー） |
| SQL を直接実行 | SQL Editor（左メニュー） |
| ログ確認 | Logs（左メニュー） |
| 認証ユーザー確認 | Authentication > Users |

### マイグレーション（DB変更）のやり方

詳しくは [8章](#8-データベース設計) と [13章](#13-よくある開発作業) を参照してください。

## 4. Vercel 詳細

### Vercel とは

**Vercel** は、Next.js アプリを簡単に公開できるホスティングサービスです。
GitHub に push するだけで自動的にビルド（コードをウェブサイト用に変換する作業）してインターネットに公開してくれます。

### Hobby プランの制限

| 項目 | 無料枠 | 備考 |
|------|--------|------|
| 帯域（データ転送量） | 100 GB/月 | ページが読み込まれた際のデータ量 |
| Serverless Functions（サーバー処理） | 100 GB-時間/月 | API 呼び出しごとに消費 |
| チームメンバー | 1人のみ | 個人利用のみ |
| デプロイ数 | 無制限 | git push のたびにデプロイ |

### Pro プランが必要なタイミング

| 状況 | 対応 |
|------|------|
| 商用利用（クライアントへの提供） | Hobby は個人非商用が前提。Pro（$20/月〜）へ移行が必要 |
| チーム開発 | 複数人が Vercel 管理画面にアクセスする場合 |
| 独自ドメインでの SSL | Hobby でも独自ドメイン自体は使えるが、本格運用は Pro 推奨 |

### デプロイの仕組み

```
コードを編集 → git commit → git push → GitHub → Vercel が自動検知 → ビルド → 公開
```

デプロイは数分で完了します。Vercel ダッシュボードの「Deployments」タブでログを確認できます。

### 環境変数の設定場所

本番環境の環境変数（APIキーなど）は **Vercel ダッシュボード** で設定します。

1. [https://vercel.com/dashboard](https://vercel.com/dashboard) にアクセス
2. プロジェクトを選択
3. Settings → Environment Variables
4. 変数名と値を入力して Save

ローカル開発では `.env.local` ファイルで設定します（[6章参照](#6-環境変数の説明)）。

### 本番 URL とプレビュー URL の違い

| URL の種類 | 説明 | 例 |
|-----------|------|-----|
| 本番 URL | `main` ブランチに push されたときに公開されるURL | `xxx.vercel.app` |
| プレビュー URL | 別ブランチに push されたときに自動生成されるURL | `xxx-git-feature-branch.vercel.app` |

プレビュー URL は「本番に影響を与えずに動作確認するための仮URL」です。

## 5. ローカル開発環境のセットアップ

このシステムを自分のパソコンで動かすための手順です。ゼロから始める場合は順番通りに実行してください。

### 前提条件

- Windows / Mac / Linux のいずれか
- インターネット接続

---

### 手順 1: 必要なツールのインストール

**Node.js**（JavaScript の実行環境）をインストールします。

1. [https://nodejs.org/](https://nodejs.org/) にアクセス
2. 「LTS」版をダウンロード・インストール
3. インストール確認：

```bash
node --version   # v20.x.x のような表示が出ればOK
npm --version    # v10.x.x のような表示が出ればOK
```

**Git**（バージョン管理ツール）をインストールします。

1. [https://git-scm.com/](https://git-scm.com/) からダウンロード・インストール
2. インストール確認：

```bash
git --version   # git version 2.x.x のような表示が出ればOK
```

---

### 手順 2: リポジトリのクローン

「リポジトリのクローン」とは、GitHub 上にあるコードを自分のパソコンにコピーすることです。

```bash
git clone https://github.com/<GitHubユーザー名>/unso-os.git
cd unso-os
```

---

### 手順 3: パッケージのインストール

```bash
npm install
```

`node_modules` フォルダが作成され、必要なライブラリがすべてインストールされます。
（数分かかる場合があります）

---

### 手順 4: 環境変数ファイルの作成

`.env.local` という名前のファイルをプロジェクトのルート（一番上のフォルダ）に作成し、以下の内容を記入します。

```
NEXT_PUBLIC_SUPABASE_URL=（Supabase管理画面のProject URLをここに貼り付け）
NEXT_PUBLIC_SUPABASE_ANON_KEY=（Supabase管理画面のanon keyをここに貼り付け）
SUPABASE_SERVICE_ROLE_KEY=（Supabase管理画面のservice_role keyをここに貼り付け）
```

各キーの取得場所：Supabase 管理画面 → プロジェクト選択 → Settings → API

> 注意：`.env.local` は Git にコミットしないでください（理由は [6章](#6-環境変数の説明) 参照）

---

### 手順 5: 開発サーバーの起動

```bash
npm run dev
```

起動すると以下のように表示されます：

```
▲ Next.js 16.x.x
- Local:        http://localhost:3000
```

ブラウザで `http://localhost:3000` にアクセスして画面が表示されれば成功です。

---

### 手順 6: デモデータの確認

Supabase 管理画面の SQL Editor から `supabase/migrations/` フォルダ内の SQL ファイルを確認できます。
デモデータを投入する場合は、Supabase SQL Editor にデータ挿入の SQL を貼り付けて実行します。

## 6. 環境変数の説明

「環境変数」とは、パスワードや接続先URLなど、コードに直接書いてはいけない設定値を外部から渡す仕組みです。

### 各変数の説明

#### `NEXT_PUBLIC_SUPABASE_URL`

- **何のための変数**: Supabase のデータベースに接続するための URL（住所）
- **例**: `https://abcdefghijklmn.supabase.co`
- **`NEXT_PUBLIC_` がついている理由**: ブラウザ（ユーザーのパソコン側）からも使うため公開されます

#### `NEXT_PUBLIC_SUPABASE_ANON_KEY`

- **何のための変数**: Supabase にアクセスするための「一般公開用のパスワード」
- **特徴**: RLS（Row Level Security）というルールによって、見せていいデータしか見えない設計のため、公開しても比較的安全
- **`NEXT_PUBLIC_` がついている理由**: ブラウザからも使うため公開されます

#### `SUPABASE_SERVICE_ROLE_KEY`

- **何のための変数**: Supabase の「管理者用のパスワード」。RLS を無効化してすべてのデータにアクセスできる
- **なぜ秘密か**: このキーが漏れると、他の会社のデータも読み書きできてしまうため、絶対に外部に公開してはいけません
- **使用場所**: サーバー側（`lib/supabase/server.ts`）のみで使用。ブラウザには絶対に渡さない

### `.env.local` を Git にコミットしない理由

コードは GitHub（インターネット上のコード保管場所）に公開されています。
`.env.local` をコミットすると、キーが全世界に公開されてしまいます。

悪意ある第三者がキーを使ってデータを盗んだり、Supabase アカウントを乗っ取ったりする可能性があるため、`.gitignore` というファイルで `.env.local` をコミット対象から除外しています。

## 7. システム全体のアーキテクチャ

「アーキテクチャ」とは、システムがどんな部品で構成され、どのように連携しているかを示す設計図のことです。

### 全体構成図

```
┌─────────────────────────────────────────────────────────┐
│                   ユーザーのブラウザ                      │
│  （Chrome / Safari / Edge など）                         │
└─────────────────┬───────────────────────────────────────┘
                  │ HTTPS（暗号化された通信）
                  ▼
┌─────────────────────────────────────────────────────────┐
│                      Vercel                              │
│  ┌───────────────────────────────────────────────────┐  │
│  │              Next.js アプリケーション               │  │
│  │                                                   │  │
│  │  ┌──────────────┐    ┌───────────────────────┐   │  │
│  │  │  ページ (UI)   │    │  API Route Handler    │   │  │
│  │  │  app/**/*.tsx │    │  app/api/**/*.ts      │   │  │
│  │  └──────────────┘    └───────────────────────┘   │  │
│  └───────────────────────────────────────────────────┘  │
└─────────────────┬───────────────────────────────────────┘
                  │ HTTPS（Supabase SDK 経由）
                  ▼
┌─────────────────────────────────────────────────────────┐
│                     Supabase                             │
│  ┌──────────────┐  ┌────────────┐  ┌────────────────┐  │
│  │  PostgreSQL   │  │    Auth    │  │    Storage     │  │
│  │  (データベース) │  │  (認証管理)  │  │  (ファイル保存) │  │
│  └──────────────┘  └────────────┘  └────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

### Google OAuth 認証フロー

「認証」とは「本人確認」のことです。このシステムでは Google のアカウントを使って本人確認します。

```
1. ユーザーが「Googleでログイン」ボタンをクリック
      ↓
2. Google の認証画面に移動
      ↓
3. Google でログイン（Googleが本人確認）
      ↓
4. Google が「OK」の証明書（認証コード）を発行
      ↓
5. /auth/callback で証明書をセッション（ログイン状態）に変換
      ↓
6. Supabase がセッションを管理（次回からは自動ログイン）
      ↓
7. ログイン完了 → トップページに移動
```

「セッション」とは「ログイン中という状態の記録」のことです。ブラウザを閉じても一定期間ログイン状態が維持されます。

## 8. データベース設計

### テーブル一覧

「テーブル」とは、Excelのシートのようなもので、データを行・列で管理する仕組みです。

| テーブル名 | 役割 | 主な内容 |
|-----------|------|---------|
| `companies` | テナント（利用企業）管理 | 会社名・プランなど |
| `employees` | 社員マスタ | ユーザー情報・ロール・会社所属 |
| `emp_requests` | 社員登録申請 | 申請内容・承認状況 |
| `customers` | 荷主マスタ | 取引先の会社情報 |
| `routes` | ルートマスタ | 運行ルートの定義 |
| `ratecards` | 運賃マスタ | ルート別の単価 |
| `vehicles` | 車両マスタ | 車両情報 |
| `billables` | 運行実績 | 実績・承認状況・請求管理 |
| `expense_categories` | 経費区分マスタ | 経費の分類定義 |
| `expenses` | 経費申請 | 経費申請・承認 |
| `attendances` | 勤怠 | 勤怠記録 |
| `monthly_closings` | 月次締め | 月ごとの締め状態管理 |
| `payrolls` | 給与 | 給与計算結果 |

### マルチテナント（複数会社のデータ分離）の仕組み

全テーブルに `company_id`（会社を識別するID）が付いており、他の会社のデータは物理的にアクセスできない設計です。

```
companies テーブル
  ├── company_id: AAA → A社のデータ
  └── company_id: BBB → B社のデータ

employees テーブル
  ├── company_id: AAA, name: 田中太郎 → A社の社員
  └── company_id: BBB, name: 鈴木花子 → B社の社員（A社からは見えない）
```

### RLS（Row Level Security）とは

RLS とは「行レベルのセキュリティ」という仕組みで、データベースが自動的に「見せていいデータだけ返す」フィルタをかけてくれます。

```sql
-- ログインユーザーが自分の会社のデータしか見えないポリシーの例
-- （自分の company_id と一致する行だけ返す）
```

`get_my_company_id()` という関数が、ログイン中のユーザーのメールアドレスから `company_id` を自動的に取得します。

### マイグレーションファイル一覧

「マイグレーション」とは「データベースの設計変更を記録したファイル」のことです。

| ファイル | 内容 |
|---------|------|
| `001_initial_schema.sql` | 全テーブルの初期定義 + RLS 基本設定 |
| `002_fix_rls_security_definer.sql` | RLS のセキュリティ修正 |
| `003_emp_requests_insert_policy.sql` | 社員申請テーブルの挿入ポリシー追加 |
| `004_storage_receipts.sql` | 領収書画像保存用ストレージ設定 |
| `005_monthly_closings.sql` | 月次締めテーブル追加 |
| `006_payroll.sql` | 給与テーブル追加 |

### マイグレーションの追加方法

DBの設計を変更したい場合は、必ず新しいマイグレーションファイルを追加します。既存ファイルは絶対に編集しないでください。

```bash
# 新しいマイグレーションファイルを作成（ファイル名は連番）
# 例: 007_add_new_column.sql を supabase/migrations/ に作成

# Supabase CLI でマイグレーションを適用
npx supabase db push
```

## 9. ロールと権限

### 各ロールができること・できないこと

| 機能 | DRIVER | ADMIN | OWNER |
|------|--------|-------|-------|
| 自分の運行実績を入力 | ✅ | ✅ | ✅ |
| 他人の運行実績を確認 | ❌ | ✅ | ✅ |
| 運行実績を承認 | ❌ | ✅ | ✅ |
| 自分の経費申請 | ✅ | ✅ | ✅ |
| 経費申請を承認 | ❌ | ✅ | ✅ |
| 自分の勤怠入力 | ✅ | ✅ | ✅ |
| 勤怠管理（全員分） | ❌ | ✅ | ✅ |
| マスタ管理 | ❌ | ✅ | ✅ |
| 社員申請の承認 | ❌ | ✅ | ✅ |
| ダッシュボード閲覧 | ❌ | 一部 | ✅ |
| 給与・請求書 | ❌ | ❌ | ✅ |

### 新しいユーザーが登録される流れ

```
1. 新しいユーザーが Google でログイン
      ↓
2. システムが employees テーブルを確認
      ↓
3. 未登録の場合 → /register（登録申請画面）に移動
      ↓
4. ユーザーが氏名・希望ロールを入力して申請
      ↓
5. emp_requests テーブルに申請が記録される（ステータス: PENDING）
      ↓
6. ADMIN / OWNER が /admin 画面で申請を確認
      ↓
7. 承認 → employees テーブルに追加（ステータス: APPROVED）
      ↓
8. ユーザーがログインできるようになる
```

## 10. 主要機能の概要

### 運行実績入力・承認フロー

ドライバーが日々の運行（どの荷主のどのルートを走ったか）を入力し、管理者が承認するフローです。

```
DRIVER: 運行実績を入力 → ADMIN: 実績を確認・承認 → 請求書作成に連携（予定）
```

### 経費申請・承認フロー

ドライバーが経費（ガソリン代・高速料金など）を申請し、管理者が承認するフローです。領収書画像を添付できます。

```
DRIVER: 金額・領収書を入力 → ADMIN: 申請を確認・承認 → 給与計算に連携（予定）
```

### 勤怠管理

出退勤の記録と確認です。DRIVER は自分の勤怠を入力し、ADMIN は全員の勤怠を管理します。

### ダッシュボード（OWNER 向け）

売上・経費・人件費などを月次でグラフ表示する経営者向けの画面です。

### マスタ管理

荷主・ルート・車両・運賃などの基本情報を登録・更新します。ADMIN / OWNER のみ操作可能です。

### 請求書・給与（実装途中）

承認済みの運行実績から請求書を自動生成する機能、および勤怠・経費から給与を計算する機能です。現在開発途中です。

## 12. デプロイ手順（本番への反映方法）

「デプロイ」とは「開発したコードを本番環境（インターネットに公開されている環境）に反映すること」です。

### 基本の流れ

```bash
# 1. 変更したファイルをステージング（コミット対象に追加）
git add <変更したファイル名>

# 2. コミット（変更を記録）
git commit -m "feat: 機能の説明"

# 3. プッシュ（GitHub に送信）
git push

# → GitHub に届くと Vercel が自動的にビルド・デプロイを開始
# → 数分後に本番 URL に反映される
```

### コミットメッセージの書き方

| プレフィックス | 使うタイミング |
|--------------|-------------|
| `feat:` | 新しい機能を追加したとき |
| `fix:` | バグを修正したとき |
| `style:` | 見た目だけを変更したとき |
| `refactor:` | 動作を変えずにコードを整理したとき |
| `chore:` | 設定ファイルの変更など |
| `docs:` | ドキュメントの変更 |

### Vercel のデプロイログの確認

1. [https://vercel.com/dashboard](https://vercel.com/dashboard) にアクセス
2. プロジェクトを選択
3. 「Deployments」タブを開く
4. 最新のデプロイをクリックしてログを確認
5. エラーがある場合は赤く表示されます

### ロールバック（前の状態に戻す方法）

```bash
# 前のコミットを確認
git log --oneline

# 指定のコミットに戻す（コミットIDを指定）
git revert <コミットID>
git push
```

または Vercel ダッシュボードの「Deployments」から古いデプロイを選択し「Redeploy」することもできます。

## 13. よくある開発作業

### 新しい機能を追加するパターン

大きな機能を追加する場合の典型的な流れです。

```
1. DBにテーブルが必要 → マイグレーション SQL を作成して適用
2. サーバー側ロジック → lib/server/ に TypeScript 関数を作成
3. API エンドポイント → app/api/*/route.ts を作成
4. 画面（UI） → app/**/page.tsx と *-page-client.tsx を作成
```

### DB テーブルを変更する方法（マイグレーション手順）

```bash
# 1. 新しいマイグレーションファイルを作成
# ファイル名: supabase/migrations/007_変更内容.sql

# 2. SQL を記述（例：カラムの追加）
# ALTER TABLE billables ADD COLUMN memo text;

# 3. Supabase CLI でマイグレーションを適用
npx supabase db push

# 4. Supabase 管理画面でテーブルを確認
```

> **注意**: 既存の SQL ファイルは絶対に編集しないでください。必ず新しいファイルで追加します。

### API を追加する方法

`app/api/` 以下にフォルダと `route.ts` を作成します。

```typescript
// app/api/billables/route.ts の例
import { NextResponse } from 'next/server'
import { createServerSupabase } from '@/lib/supabase/server'

// GET: データを取得する API
export async function GET() {
  const supabase = await createServerSupabase()
  const { data, error } = await supabase.from('billables').select('*')
  if (error) return NextResponse.json({ error: error.message }, { status: 500 })
  return NextResponse.json(data)
}
```

### UI コンポーネントを追加する方法

shadcn/ui（既製のデザインコンポーネント集）から新しいコンポーネントを追加します。

```bash
# 例：Accordion（折りたたみパネル）を追加
npx shadcn@latest add accordion
```

追加されたコンポーネントは `components/ui/` に保存されます。

## 14. コーディング規約

### 基本ルール

| 項目 | ルール |
|------|--------|
| 言語 | TypeScript（型が必須。型なしは NG） |
| コンポーネント基本 | React Server Components（サーバーで実行） |
| 操作が必要な部分 | `'use client'` を先頭に付けて Client Components にする |
| スタイル | Tailwind CSS を優先。独自 CSS は最小限 |
| UI 部品 | shadcn/ui を優先使用 |
| API | Next.js Route Handler（`app/api/*/route.ts`） |
| DB アクセス | Supabase JS クライアント経由のみ |
| エラー処理 | try-catch で囲み、ユーザーにわかるメッセージを返す |
| 関数の長さ | 1関数30行以内を目安 |
| コメント | 日本語。関数の先頭に「何をする関数か」を1行で記載 |

### ファイル命名規則

| ファイルの種類 | 命名規則 | 例 |
|-------------|---------|-----|
| ページ | `page.tsx` | `app/dashboard/page.tsx` |
| クライアントページ | `*-page-client.tsx` | `dashboard-page-client.tsx` |
| API | `route.ts` | `app/api/billables/route.ts` |
| サーバーロジック | `*.ts` | `lib/server/billable.ts` |
| shadcn コンポーネント | `*.tsx` | `components/ui/button.tsx` |
| 機能コンポーネント | `*.tsx` | `components/features/BillableForm.tsx` |

### Server Components vs Client Components

```
Server Components（サーバー側で実行）
  → データ取得、認証チェック、初期表示
  → ページファイル（page.tsx）はほぼこれ

Client Components（ブラウザで実行）
  → ボタンクリック、フォーム入力、状態管理
  → 先頭に 'use client' を記述
  → *-page-client.tsx はほぼこれ
```

## 15. トラブルシューティング

### よくあるエラーと対処法

#### ローカルで起動できない

```bash
# エラー: Cannot find module 'xxx'
npm install    # パッケージが足りない場合

# エラー: .env.local が見つからない
# → .env.local を作成して環境変数を設定（6章参照）
```

#### ローカルで動くが本番で動かない

1. Vercel ダッシュボードで「Deployments」のエラーログを確認
2. 環境変数が Vercel に設定されているか確認（Settings → Environment Variables）
3. ビルドエラーの場合：`npm run build` をローカルで実行してエラーを再現

#### 認証がうまくいかない場合

- Supabase 管理画面 → Authentication → Providers → Google が有効か確認
- Google OAuth の「承認済みのリダイレクト URI」に Vercel の URL が登録されているか確認
- ローカル開発なら `http://localhost:3000/auth/callback` も登録が必要

#### Supabase の RLS でデータが取れない場合

症状：データを取得しようとすると空配列が返ってくる（エラーではない）

原因：RLS ポリシーによってデータへのアクセスがブロックされている

対処法：

1. Supabase 管理画面 → SQL Editor で以下を実行してポリシーを確認

```sql
SELECT * FROM pg_policies WHERE tablename = 'billables';
```

2. `employees` テーブルに自分のメールアドレスが登録されているか確認

```sql
SELECT * FROM employees WHERE google_email = '自分のメールアドレス';
```

3. `get_my_company_id()` 関数が正しく動作するか確認

```sql
SELECT get_my_company_id();
```
