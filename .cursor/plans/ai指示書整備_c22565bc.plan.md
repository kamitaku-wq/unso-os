# AI指示書と実装計画の構築

## 前提

- 現行の土台は、Googleログイン、Supabase接続、初期DBスキーマ、会社単位のRLSまで。根拠ファイル: [app/login/page.tsx](C:/Users/kamit/dev/unso-os/app/login/page.tsx), [app/auth/callback/route.ts](C:/Users/kamit/dev/unso-os/app/auth/callback/route.ts), [proxy.ts](C:/Users/kamit/dev/unso-os/proxy.ts), [supabase/migrations/001_initial_schema.sql](C:/Users/kamit/dev/unso-os/supabase/migrations/001_initial_schema.sql)
- 現状整理の基準文書は [docs/implementation-status.md](C:/Users/kamit/dev/unso-os/docs/implementation-status.md)
- 旧システムの正解源は単独では固定せず、旧GAS・現リポジトリ・都度判断を組み合わせ、食い違いはケースごとに確認する
- 対象はMVPではなく SaaS 全体。ただし監査・証跡は初期段階では最小限、AI指示書は「厳格すぎない実務ガイド」寄りで作る
- 当面から他業種展開を見据えるため、運送業固有用語を保ちつつ、設定で抽象化できる境界を明記する

## 作る成果物

- AI向け指示書: AIが実装時に守る判断基準、優先順、禁止事項、確認必須条件をまとめる
- 実装計画書: フェーズ、依存関係、未確定事項、実装順、確認ポイントをまとめる

## 収集・整理する論点

- 開発環境整備: Node/npm、環境変数、Supabase接続、ローカル起動、初期データ投入、テスト方法
- デプロイ運用: Vercel前提、環境変数の管理、公開前チェック、バックアップと障害時の切り戻し方針
- 社員導線: 初回会社作成、初回オーナー作成、社員申請、承認待ち、利用開始の流れ
- マルチテナント: `company_id` 分離、RLS責務、会社作成時の初期データ、他業種展開との両立
- マスタ設計: 荷主、ルート、車両、運賃、経費区分、将来の業種別設定の切り分け
- AIルール: どこまで自動判断してよいか、いつ確認するか、編集禁止対象、説明粒度、テスト・コミットの扱い

## AI向け指示書の章立て案

- プロダクト前提: 何を作るか、誰が使うか、現時点で未実装な範囲
- 判断ルール: 競合する仕様が出た時の扱い、ケースごと確認すべき分岐
- 実装原則: 画面・API・RLSの3層をそろえる、ローカルで再現できる順に作る、他業種展開を壊さない抽象化
- 確認必須項目: 認証、社員導線、会社作成、マスタ初期値、外部サービス接続、環境変数
- 禁止/注意事項: `.env.local` 非コミット、既存マイグレーションを直接改変しない、曖昧な権限制御を推測で実装しない

## 実装計画書の骨子

- Phase 0: 開発環境と外部サービスの可視化
  - ローカル起動条件、Supabase/Vercel/Google OAuthの保有権限、必要シークレット一覧を確定
- Phase 1: 認証後導線と社員導線の確定
  - ログイン後の遷移先、初回ユーザーの扱い、社員申請/承認フロー、最初の会社とオーナー作成
- Phase 2: マルチテナントとマスタ責務の整理
  - `company_id` とRLSの責務分担、共通マスタと会社別マスタの境界、他業種化の設定方式
- Phase 3: 中核業務MVPの選定
  - `運行実績 / 経費 / 勤怠 / 社員導線 / 基盤` の中から、依存関係ベースで最初の実装対象を決定
- Phase 4: デプロイ運用の最小設計
  - 環境変数、反映手順、最低限の障害対応、公開前チェック項目を定義

## 現時点で優先度が高い判断

- 最優先は「ログイン後に誰が何をできるか」の定義。現状 `/` は仮ページのため、業務開始地点がない
- 次は `employees` と Google アカウントの結び付け、および最初の会社/最初のオーナーをどう作るか
- その次に、他業種展開を意識しつつ、どの業務を最初の実装対象にするかを依存関係ベースで選定する
